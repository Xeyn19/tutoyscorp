# TutoY Corp Integrated System

Marketing site, inquiry flow, and admin dashboard for `TutoY Corp Integrated System`, built with Next.js App Router, React, Tailwind CSS, Supabase, Cloudflare Turnstile, Nodemailer, React Toastify, and Recharts.

## Current Scope

- landing page with reusable content-driven sections
- target market section focused on transport companies, tourism agencies, and booking services
- efficiency proof bar chart comparing manual workflows against TutoY Corp System processing time
- dedicated contact page with inquiry submission
- admin login with Supabase email/password auth
- protected dashboard shell with fixed sidebar and shared layout
- overview page with live inquiry stats and chart filters
- records page with searchable data table and message modal
- Cloudflare Turnstile verification before inquiry submission
- Supabase-backed storage for contact inquiries
- confirmation email sending through Gmail SMTP / Nodemailer
- toast feedback for inquiry submit, login success, and logout success
- light and dark theme support with persistent theme preference

## Local Setup

Install dependencies:

```bash
npm install
```

Create `.env.local` in the project root:

```env
NEXT_PUBLIC_SUPABASE_URL=https://YOUR_PROJECT_ID.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=YOUR_PUBLISHABLE_OR_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY=YOUR_SERVICE_ROLE_KEY

NEXT_PUBLIC_TURNSTILE_SITE_KEY=YOUR_TURNSTILE_SITE_KEY
TURNSTILE_SECRET_KEY=YOUR_TURNSTILE_SECRET_KEY

GMAIL_SMTP_USER=your-account@gmail.com
GMAIL_SMTP_PASS=your-gmail-app-password
MAIL_FROM="TutoY Corp Integrated System <your-account@gmail.com>"
MAIL_REPLY_TO=your-account@gmail.com
```

Notes:

- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` is the preferred browser key.
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` also works as a fallback if you still use the older naming.
- `SUPABASE_SERVICE_ROLE_KEY` must stay server-side only.
- After editing `.env.local`, restart `npm run dev`.

Start the dev server:

```bash
npm run dev
```

Open `http://localhost:3000`.

## Route Map

- `/` main landing page
- `/landingpage` alternate route for the same landing experience
- `/contact` contact and inquiry form page
- `/login` admin login page
- `/dashboard` protected overview page with live stats and chart
- `/dashboard/records` protected records page with searchable inquiry table
- `/api/inquiries` POST endpoint for Turnstile verification, Supabase insert, and confirmation email

## Dashboard Structure

The admin area is split into two real routes, not in-page scroll sections:

- `Overview` at `/dashboard`
  - total inquiries
  - unique companies
  - dynamic chart with `Daily`, `Weekly`, `Monthly`, and `Custom Range` filters
- `Records` at `/dashboard/records`
  - search by name, company, email, contact number, service, or message
  - full-width data table
  - clickable message cell that opens a modal

Both pages use the shared dashboard shell in `src/app/dashboard/layout.jsx` and `src/components/dashboard/DashboardShell.jsx`.

## Supabase Setup

This project uses two Supabase layers:

1. Supabase Auth for admin email/password sign-in
2. Supabase tables for inquiry records and the admin allowlist

### 1. Create the inquiry table

Run this in the Supabase SQL Editor:

```sql
create table if not exists public.contact_inquiries (
  id bigint generated by default as identity primary key,
  full_name text not null,
  email text not null,
  contact_number text not null,
  company_name text,
  selected_service text not null,
  message text not null,
  consent_accepted boolean not null default false,
  created_at timestamptz not null default timezone('utc', now())
);
```

### 2. Create the admin allowlist table

```sql
create table if not exists public.admin_users (
  email text primary key,
  created_at timestamptz not null default timezone('utc', now())
);
```

### 3. Create the admin user

Do not create a password column in `admin_users`.

Use Supabase dashboard:

1. Go to `Authentication -> Users`
2. Add a user with email and password
3. Insert the same email into `public.admin_users`

Example:

```sql
insert into public.admin_users (email)
values ('admin@example.com')
on conflict (email) do nothing;
```

Access rules:

- if the user is not authenticated, `/dashboard` redirects to `/login`
- if the user is authenticated but not in `admin_users`, access is denied
- only allowlisted emails can open `/dashboard` and `/dashboard/records`

## Inquiry Flow

1. `src/app/contact/page.jsx` renders the inquiry UI.
2. `src/components/landingpage/InquiryForm.jsx` collects the form data and Turnstile token.
3. `src/app/api/inquiries/route.js` validates the payload and verifies Turnstile.
4. The inquiry is inserted into `public.contact_inquiries`.
5. `src/lib/inquiry-mail.js` sends the confirmation email.
6. The user receives toast feedback for loading, success, or failure.

## Notes

- Toast notifications are configured globally and currently display in the top-right corner.
- The login page handles missing Supabase browser config gracefully and shows a user-facing message instead of crashing.
- `contact_inquiries` is queried server-side for the dashboard.
- The landing page efficiency chart uses curated static data in `src/data/landingpage-content.js`, not runtime Excel parsing.
- The dashboard shell uses the shared mascot logo from `public/tutoy-logo.jpeg`.
- Lint currently passes with one existing warning in `src/components/landingpage/BrandMark.jsx` for using `<img>` instead of `next/image`.
- See `dashboard-auth.md` for a shorter admin setup guide focused on Supabase login provisioning.
